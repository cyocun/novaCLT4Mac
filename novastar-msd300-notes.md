# NovaStar MSD300 macOSアプリ開発 - 調査メモ

## 目的
- MSD300をmacOSから制御するSwiftネイティブアプリを作る
- 機能: 輝度調整 + レイアウト設定のみ

## 調査結果

### デバイス情報
- **接続方式**: USB (Silicon Labs CP210x USB-UART変換)
- **VID**: 0x10C4 (Silicon Labs)
- **PID**: 0xEA60 (CP210x USB to UART Bridge)
- **シリアル**: D4A93018614CEE11878917890D31E3D9
- **通信方式**: USBシリアル (COMポート経由のUART通信)
- **接続**: USB + HDMI両方接続済み

### 重要な発見
- MSD300はUSB生パケットではなく、USB-シリアル変換チップ経由で通信
- つまりNovaLCTはCOMポート経由でシリアルプロトコルでMSD300を制御
- macOSアプリではIOKitまたはORSSerialPortで同様のシリアル通信を実装可能

### COMポート情報
- **COMポート**: COM4
- **ドライバ**: Silicon Labs CP210x USB to UART Bridge
- **ボーレート**: 115200
- **データビット**: 8
- **パリティ**: None
- **ストップビット**: 1
- **フロー制御**: なし (XON/XOFF OFF, CTS OFF, DSR OFF)

### USBPcapキャプチャ情報
- **USBPcapインターフェース**: USBPcap1
- **デバイスアドレス**: 1.7.0 (デバイス番号7)
- **ルートハブ**: USB\ROOT_HUB30\5&1026d542&0&0
- **デバイスツリー**: ROOT_HUB → VID_2109 (ハブ) → VID_1A40 (ハブ) → VID_10C4 (CP210x/MSD300)

## プロトコル解析結果

### パケット構造（初期解析）
- **host→device (コマンド)**: `55 AA` で開始
- **device→host (応答)**: `AA 55` で開始
- **エンドポイント**: 1.7.2 (Bulk転送)

### パケットフォーマット（暫定）
```
55 AA [seq_hi seq_lo] FE [dest] [addr 4bytes] [offset 4bytes] [length 2bytes] [data...] [checksum 2bytes]
```

### 観測されたパケットタイプ
1. **20バイトコマンド (読み出し)**: データペイロードなし
   例: `55aa00e8fe000000000000000200000002003f57`

2. **21バイトコマンド (1バイト書き込み)**:
   例: `55aa00fafe0001ffffff01000100000201000c5c5a`
   → アドレス `01ffffff0100`, オフセット `01000002`, データ `01`

3. **24バイトコマンド (4バイト書き込み)**:
   例: `55aa00fbfe0001ffffff0100e30100020400f0f0f000075e`
   → データ `f0f0f0 00` — 色/ガンマ関連の可能性

4. **大量データ応答 (64バイト×複数)**:
   → ガンマテーブルまたは設定データの読み出し

### 輝度コマンド（確定）
**21バイト書き込みコマンドが輝度設定であることを確認**

コマンド構造:
```
55 AA [seq_hi] [seq_lo] FE 00 01 FF FF FF 01 00 01 00 00 02 01 00 [brightness] [chk_hi] [chk_lo]
```

- **ヘッダ**: `55 AA` (固定)
- **シーケンス番号**: 2バイト、送信ごとに+2ずつ増加
- **コマンドタイプ**: `FE` (書き込み)
- **デバイスアドレス**: `00 01 FF FF FF 01 00` (MSD300宛)
- **レジスタオフセット**: `01 00 00 02` (輝度レジスタ)
- **データ長**: `01 00` (1バイト)
- **輝度値**: `00`(0%)〜`FF`(100%) ← ここが輝度
- **チェックサム**: 2バイト、リトルエンディアン（アルゴリズム解析済み、下記参照）

検証データ:
| 輝度値(hex) | 輝度値(dec) | キャプチャでの確認 |
|------------|------------|-------------------|
| 0x00       | 0          | ✓ 最小値で確認     |
| 0x09       | 9          | ✓ 低い値で確認     |
| 0xFF       | 255        | ✓ 最大値で確認     |

**応答フォーマット** (device→host, 20バイト):
```
AA 55 [seq_hi] [seq_lo] 00 FE 01 FF FF FF 01 00 01 00 00 02 00 00 [chk_hi] [chk_lo]
```

### 付随コマンド（色/ガンマ設定）
輝度コマンドの直後に毎回送信される24バイトコマンド:
```
55 AA [seq] FE 00 01 FF FF FF 01 00 E3 01 00 02 04 00 F0 F0 F0 00 [checksum]
```
- レジスタ: `E3 01 00 02` (色/ガンマレジスタ)
- データ: `F0 F0 F0 00` (RGB=240,240,240 — 輝度変更では固定値)

### ポーリングパターン
- NovaLCTは接続中に定期的にMSD300をポーリング
- 輝度コマンドもポーリング周期で繰り返し送信される
- 設定データ・ガンマテーブルを周期的に読み出し

### チェックサムアルゴリズム（解析完了）
```
checksum = (0x5555 + sum(全バイト、55AAヘッダと末尾チェックサム2バイトを除く)) & 0xFFFF
```
- **シード値**: `0x5555`
- **対象**: ヘッダ `55 AA` の後ろから、チェックサムの直前まで全バイトの合計
- **格納順**: リトルエンディアン `[checksum & 0xFF, (checksum >> 8) & 0xFF]`

検証:
| パケット | バイト合計 | 0x5555+合計 | 格納値(LE) | 実際のチェックサム | 一致 |
|---------|-----------|------------|-----------|-----------------|------|
| seq=0040,bri=09 | 0x044A | 0x599F | 9F,59 | 9F,59 | ✓ |
| seq=0042,bri=FF | 0x0542 | 0x5A97 | 97,5A | 97,5A | ✓ |
| seq=0058,bri=00 | 0x0359 | 0x58AE | AE,59... | AE,59 | ✓ |
| 20byte read cmd  | 0x01EA | 0x573F | 3F,57 | 3F,57 | ✓ |
| 24byte color cmd | 0x08B2 | 0x5E07 | 07,5E | 07,5E | ✓ |

### レイアウト設定コマンド（解析完了）

レイアウト変更（Screen Configuration → Send to HW）は単純なレジスタ書き込みではなく、
**~50+コマンドのシーケンス**を送信する複雑な操作であることが判明。

#### キャプチャ条件
- **layout3.pcap**: 4列×1行構成で「右→左」と「左→右」のスキャン方向を変更
- NovaLCT の Screen Configuration で方向変更後「Send to HW」ボタンで書き込み

#### レイアウトコマンドシーケンスの構造

1. **全体設定コマンド群** (~10コマンド)
   - 送信元→MSD300 (dest=0x00): 画面全体の解像度、スキャン方向等
   - スキャン方向レジスタ: `0x01000088`
   - 画面幅レジスタ: `0x02000002`
   - 画面高さレジスタ: `0x02000003`

2. **受信カード個別設定** (カード数×数コマンド)
   - 送信先を受信カード(dest=0xFF)に変更
   - ポート番号でカードを指定
   - カードごとの開始座標、サイズ等を設定

3. **マッピングテーブル書き込み** (16ブロック×276バイト)
   - 各ブロック276バイトの大きなデータブロック
   - LEDモジュール→ピクセル座標のマッピング情報
   - スキャン方向によってデータ内容が変わる

#### 右→左 vs 左→右 の違い

| 項目 | 左→右 (L→R) | 右→左 (R→L) |
|------|-------------|-------------|
| カード送信順 | 0, 1, 2, 3 | 3, 2, 1, 0 |
| マッピングテーブル | パターンA | パターンB（反転） |
| スキャン方向バイト | 異なる値 | 異なる値 |

#### 実装方針: プリセット方式
レイアウトコマンドは非常に複雑なため、以下のアプローチを採用:

1. **プリセット定義**: よく使うレイアウトパターンをキャプチャしたコマンドシーケンスとして保存
2. **リプレイ方式**: 保存済みコマンドシーケンスのシーケンス番号とチェックサムを再計算して送信
3. **対応パターン**:
   - 4列×1行 左→右 (キャプチャ済み: `captures/layout_4x1_left_to_right.txt`)
   - 4列×1行 右→左 (キャプチャ済み: `captures/layout_4x1_right_to_left.txt`)
   - 2列×4行 S字パターン (キャプチャ済み: `captures/layout_2x4_serpentine.txt`)
   - 1列×4行 上→下 (追加キャプチャ予定)

#### 送信先バイト (dest) の使い分け
- `dest=0x00`: MSD300本体（送信カード）宛のコマンド
- `dest=0xFF`: 受信カード宛のコマンド（レイアウト設定で使用）

### パケットフォーマット（確定版）

```
55 AA [seq_hi] [seq_lo] FE [dest] [device_type] [port] [board_hi] [board_lo] [direction] [reserved] [reg_0] [reg_1] [reg_2] [reg_3] [len_lo] [len_hi] [data...] [chk_lo] [chk_hi]
```

| オフセット | サイズ | 名前 | 説明 |
|-----------|--------|------|------|
| 0-1 | 2 | header | `55 AA` (固定) |
| 2-3 | 2 | sequence | シーケンス番号 (送信ごとに+1) |
| 4 | 1 | source | `FE` = PC |
| 5 | 1 | dest | `00` = 送信カード, `FF` = 受信カード |
| 6 | 1 | device_type | `01` = 受信カード |
| 7 | 1 | port | ポート番号 (`FF` = 全ポート) |
| 8-9 | 2 | board_index | ボードインデックス (`FF FF` = 全ボード) |
| 10 | 1 | direction | `01` = 書き込み, `00` = 読み取り |
| 11 | 1 | reserved | `00` |
| 12-15 | 4 | register | レジスタアドレス (リトルエンディアン) |
| 16-17 | 2 | length | データ長 (リトルエンディアン) |
| 18+ | N | data | データペイロード (書き込み時のみ) |
| 末尾2 | 2 | checksum | チェックサム (リトルエンディアン) |

### レジスタマップ（判明分）

| レジスタ | アドレス | サイズ | 説明 |
|---------|---------|--------|------|
| globalBrightness | 0x02000001 | 1 | 全体輝度 (0x00-0xFF) |
| screenWidth | 0x02000002 | 2 | 画面幅 (ピクセル) |
| screenHeight | 0x02000003 | 2 | 画面高さ (ピクセル) |
| scanDirection | 0x01000088 | 1 | スキャン方向 |
| testPattern | 0x02000101 | 1 | テストパターン |
| rgbBrightness | 0x020001E3 | 4 | RGB個別輝度 (R,G,B,0x00) |

#### 2×4 S字パターンの特徴
- **layout_vertical.pcap**: 2列×4行構成、S字パターン（蛇行）スキャン
- **レイアウトコマンド**: 56コマンド (4×1の43-45コマンドより多い)
- **受信カード**: 8枚（ポート0-7）
- **ポート送信順**: 3→2→1→0→4→5→6→7（S字パターン特有の順序）
- **マッピングテーブル**: 16ブロック（4×1と同数）だがデータパターンが異なる
- **受信カード保存**: キャプチャ後半（225-245s）に各カードへの設定書き込み操作も含まれる
